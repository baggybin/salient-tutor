## 2. Unify WS + CLI submit paths (telemetry + preamble)

**Goal:** WS turns record `agent_runs` telemetry. The "SERVER SESSION STATE" preamble exists in one place. Both paths call a shared `daemon._submit_turn()`.

### Files & changes

**`src/salient_tutor/daemon.py`**
- Extract a shared `_submit_turn(agent, message, *, session_id, timeout)` method:
  ```python
  async def _submit_turn(self, agent, message, *, session_id=None, timeout=120.0):
      """Shared submit path for both WS and CLI/programmatic turns."""
      if session_id:
          session = self.get_session(session_id)
          message = (
              f"SERVER SESSION STATE: phase={session['phase']}; skill_id={session['skill_id']}; "
              "Only structured assessment actions advance the lesson.\n\n" + message
          )
      runner = self._make_runner(agent)
      store = self.lesson_store
      run = (await asyncio.to_thread(store.start_agent_run, session_id, agent, str(runner.cfg), "prompt")
             if store is not None else None)
      loop = asyncio.get_running_loop()
      fut = loop.create_future()
      runner.submit(message, future=fut, max_turns_hint=runner.cfg.get("max_turns"))
      return {"future": fut, "run": run, "store": store}
  
  async def _finish_turn_run(self, handle, output, error=None):
      """Record agent_runs telemetry when the turn resolves."""
      run = handle["run"]
      if run is None:
          return
      status = "failed" if error else "completed"
      await asyncio.to_thread(handle["store"].finish_agent_run, run["run_id"], status, output=output, error=error)
  ```
- Refactor `prompt()` to delegate to `_submit_turn` + `_finish_turn_run`.
- Unify preamble wording: use **"Only structured assessment actions advance the lesson."** (drop the "not prose" variant ? it?s redundant, the "only" already says it).

## Implementation notes (resolved during build)

Four gaps the original plan above missed, found by reading the actual code
before implementation and resolved as follows:

1. **`create_session` positional INSERT would break** when the `mastery_stage`
   column was dropped â€” it used `INSERT INTO sessions VALUES (?,?,...x16)` with
   the column in position 14. Resolved: the v2->v3 migration drops the column
   AND `create_session` + the `CREATE TABLE` DDL were updated to the 15-column
   shape. `get_session()` now injects a **computed** `mastery_stage` (via
   `derive_mastery_stage` over the latest attempt) so the session dict keeps its
   contract â€” the one reader (`tests/test_lesson_store.py` assert on
   `result["session"]["mastery_stage"]`) and any future consumer still work.

2. **`analytics()` GROUP BY** moved off the column to a single correlated-subquery
   join (`sessions` LEFT JOIN latest-attempt-per-session), bucketing the derived
   stage in Python. Bulk (one query), no N+1. `get_session()` stays lean for the
   WS preamble path which only reads `phase`/`skill_id`.

3. **`_submit_turn` <-> `wait_done` plumbing**: `wait_done` grew a `run_handle`
   param. Telemetry is recorded right after `result_job = await fut` and BEFORE
   the judge filter (the agent_run records the raw LLM result; the judge's
   revised text is a separate, learner-facing concern). `_submit_turn` returns
   `{future, run, store, job}` so the WS handler keeps its `job.id` tracking.

4. **Transaction ownership**: `record_attempt_and_transition` owns the connection
   internally (not passed in) â€” `LessonController.record_attempt` pre-generates
   the `attempt_id`, calls the store method, then re-reads the attempt. The
   sqlite3 `Connection` context manager rolls back the whole turn on any failure.

### curricula_list (item 1) â€” 2-stage -> 3-stage, KG-derived
The KG gradebook has no concept of "remediation_queued" (that's a per-session
attempt outcome), so curriculum topics map to three stages off the KG:
`unstarted` (no fact) / `provisional_mastery` (studied, mastery < threshold) /
`durable_mastery` (mastery >= STRONG_THRESHOLD). `mastered_modules` still keys
on `durable_mastery`.
